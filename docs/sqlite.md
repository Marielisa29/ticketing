Q1 — InMemoryTicketRepository implémente les méthodes save, get_by_id et list_all. Ces opérations couvrent la sauvegarde (création ou mise à jour) d'un ticket, la récupération d'un ticket par son identifiant et l'énumération de tous les tickets présents en mémoire.

Q2 — Le fichier ports/ticket_repository.py définit exactement le même contrat : les méthodes abstraites save, get_by_id et list_all. L’interface décrit les signatures et le comportement attendu, et InMemoryTicketRepository fournit une implémentation concrète conforme à ce contrat.

Q3 — Si l'on souhaite remplacer InMemory par SQLite dans un use case, il ne faut pas modifier le code du use case lui‑même ni la logique métier qu’il contient. Seule l’instanciation du repository change, puisque les use cases dépendent de l’interface TicketRepository et non d’une implémentation concrète. En pratique, on remplace l’objet passé au use case (par injection de dépendance ou via une fixture dans les tests) par une instance de SQLiteTicketRepository ; les tests peuvent nécessiter une adaptation de leur setup (par exemple utiliser une fixture sqlite_ticket_repo à la place de l’inmemory), mais le corps des tests et le use case restent inchangés.
